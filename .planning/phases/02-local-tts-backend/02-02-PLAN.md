<task type="auto">
  <name>Task 1: Create LocalTTSError type with 6 failure mode classifications</name>
  <files>internal/xuezh/audio/local.go</files>
  <action>
Add a custom error type to `audio/local.go`:

```
type LocalTTSError struct {
    Reason  string // "server_down", "stale_pid", "oom", "timeout", "port_conflict", "text_too_long"
    Message string // Human-readable description
    Details map[string]any // Structured details for the error envelope
}
func (e LocalTTSError) Error() string { return fmt.Sprintf("BACKEND_FAILED:%s:%s", e.Reason, e.Message) }
```

**Update LocalTTS to return LocalTTSError instead of plain errors for these 6 modes:**

1. **server_down**: readPortFile returns 0, or readPIDFile returns 0, or connection refused on HTTP call. Details: `{port, pid}`.
2. **stale_pid**: PID file exists but processAlive returns false. Details: `{pid, port}`. Call cleanStateFiles.
3. **oom**: Server returns HTTP 500 with body containing "memory" or "OOM" or "killed", or process dies during request (connection reset). Details: `{port, model, http_status, body_preview}`.
4. **timeout**: HTTP client deadline exceeded (context.DeadlineExceeded or net/http timeout). Details: `{port, timeout_seconds}`.
5. **port_conflict**: Connection refused but PID alive — server crashed/restarted on different port, or another process took the port. Details: `{port, pid}`.
6. **text_too_long**: Server returns HTTP 400 or 422 with body suggesting input too long. Details: `{port, text_length, http_status, body_preview}`.

**Classification logic in LocalTTS (replace the simple error returns):**
- Before HTTP call: check PID alive and port file — classify as server_down or stale_pid.
- On HTTP error: read response body (limit to 1KB for body_preview), inspect status code and body content to classify oom/timeout/text_too_long/port_conflict.
- On connection refused: if PID alive → port_conflict, if PID dead → stale_pid.
- On timeout: classify as timeout.
- Default unclassified HTTP errors: use "server_down" as fallback with full details.

**Keep ffmpeg errors as-is** (ToolMissingError, ProcessFailedError) — those are post-download conversion errors, not TTS server errors.

  <name>Task 2: Wire LocalTTSError into CLI handler for structured JSON error envelopes</name>
  <files>internal/xuezh/cli/cli.go</files>
  <action>
Update the "local" backend error handling in `runAudioTTS`:

1. Add an `errors.As(err, &localTTSErr)` check (where localTTSErr is `audio.LocalTTSError`) before the existing ToolMissingError/ProcessFailedError checks.
2. When matched, emit `BACKEND_FAILED` error type with details merged from `localTTSErr.Details` plus the standard fields (text, voice, out, backend).
3. Include `localTTSErr.Reason` as "reason" in the details map so the agent can switch on it.

**Result: The agent receives JSON like:**
```json
{
  "ok": false,
  "error": {
    "type": "BACKEND_FAILED",
    "message": "BACKEND_FAILED:server_down:server not running",
    "details": {
      "reason": "server_down",
      "port": 0,
      "pid": 0,
      "text": "你好",
      "voice": "Vivian",
      "out": "...",
      "backend": "local"
    }
  }
}
```

The agent can read `details.reason` to decide action:
- server_down → run `audio server start`
- stale_pid → run `audio server start` (auto-cleans stale)
- oom → alert user, suggest smaller model
- timeout → retry or alert user
- port_conflict → run `audio server stop` then `start`
- text_too_long → split text and retry
